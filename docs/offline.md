# Офлайн-режим — план реализации

## Цель

Полноценная работа без сети: просмотр и редактирование транзакций, управление кошельками.
Не поддерживается офлайн: регистрация, вход, инвайты, управление участниками.

## Архитектурные решения

### Client-generated UUID
Все новые сущности (кошельки, транзакции) получают UUID на клиенте до отправки на сервер.
Сервер принимает переданный ID. Устраняет проблему reconciliation локальных и серверных ID.

### Queue-first запись
Запись всегда идёт через queue (не «попробуй сервер → при ошибке в queue»):
1. Запись → сначала в IDB queue + оптимистичный UI
2. Параллельно пробуем выполнить немедленно (если онлайн)
3. Успех → убираем из queue, обновляем IDB-зеркало
4. Ошибка → остаётся в queue, выполнится при reconnect

### Идемпотентность сервера
`INSERT ... ON CONFLICT (id) DO NOTHING` для create-операций.
Update и Delete идемпотентны по природе. Повторная отправка очереди безопасна.

### /api/sync вместо Server Actions для очереди
Server Actions привязаны к деплою (Next-Action ID). При обновлении приложения старые ID протухают.
`POST /api/sync` принимает batch мутаций, вызывает те же `lib/`-функции. Стабилен между деплоями.

## Слои реализации

### Слой 1 — `lib/idb.ts`
IndexedDB с версионированием схемы (onupgradeneeded).

Сторы:
- `wallets` — `{ id, name, role, createdAt }`
- `transactions` — все транзакции по walletId
- `queue` — `{ id, type, payload, createdAt }`, строго упорядочена
- `tombstones` — ID локально удалённых сущностей (чтобы не перезаписывать данными с сервера)

Функции: `saveWallets`, `getWallets`, `saveTransactions`, `getTransactions`,
`getStats` (агрегация прямо из IDB — дублирует логику `lib/stats.ts`),
`enqueue`, `getQueue`, `dequeue`, `addTombstone`, `getTombstones`.

### Слой 2 — `app/api/sync/route.ts`
`POST /api/sync` — принимает `{ items: QueueItem[] }`, выполняет последовательно.
Обрабатывает ошибки поштучно: при провале одного элемента — останавливается, возвращает индекс и причину.

QueueItem types: `addTransaction`, `updateTransaction`, `deleteTransaction`,
`createWallet`, `renameWallet`, `deleteWallet`.

### Слой 3 — lib + Server Actions: client-generated ID
`createTransaction(userId, walletId, data, id?)` — `id ?? crypto.randomUUID()`.
`createWallet(userId, name, id?)` — аналогично.
Drizzle: `.onConflictDoNothing()` для insert.

### Слой 4 — SW: кеш страниц для офлайн-навигации
Navigate-запросы: network-first + сохраняем успешный ответ в кеш.
Офлайн-навигация → отдаём закешированный HTML → JS монтируется и грузит данные из IDB.

### Слой 5 — `hooks/useOfflineSync.ts`
Слушает `online`/`offline` события.

При `online`:
1. `POST /api/sync` с queue (строго последовательно)
2. При 401 → очищаем queue, редирект на логин
3. При частичном провале → останавливаемся, показываем ошибку с деталями
4. Успех → очищаем queue + tombstones → фетчим свежие данные с сервера → обновляем IDB

Возвращает: `isOffline`, `pendingCount`, `syncError`.

### Слой 6 — Компоненты: гибридный режим
`WalletView`, `WalletSidebar`:
- При монтировании онлайн → сохраняем полученные данные в IDB
- При офлайн или пустом ответе → читаем из IDB
- При пустом IDB (первый офлайн) → явный стейт «нет данных офлайн»
- Мутации через queue-first (см. выше)
- Tombstones применяются при чтении из IDB (фильтруем удалённые)

### Слой 7 — UI
- Офлайн-индикатор (баннер или иконка)
- Счётчик непросинканных операций
- Disabled для: создание инвайт-ссылки, управление участниками кошелька

## Известные ограничения

- **Первый офлайн**: данные недоступны, пока хотя бы раз не был онлайн на этом устройстве
- **JWT истёк офлайн**: при синке получим 401, queue очищается, требуется повторный вход
- **Совместные кошельки**: при параллельных изменениях с другого устройства — last-write-wins
- **iOS PWA**: Background Sync API отсутствует; синк происходит только при открытии приложения
- **IDB schema upgrades**: при изменении структуры IDB в будущих версиях нужна миграция через onupgradeneeded
