# Лог решений

## Платформа: PWA на Next, а не Telegram Mini App / нативка

- Next максимально раскрывает React 19 (Server Actions, RSC, useOptimistic).
- PWA даёт «мобилко-подобный» опыт без App Store и React Native.
- Telegram отложен: добавим позже как ещё один клиент поверх lib/.

## Бот: чат-бот, не Mini App (когда дойдём)

- Чат-бот дешевле допилить: нет UI, только парсинг → вызов lib → ответ.
- Удобнее в быту для ввода («кофе 350» на ходу).
- Условие безболезненной миграции: вся логика в lib/, userId с первого дня.

## ORM: Drizzle

- TS-first, изучаем новое после Prisma.

## БД: PostgreSQL → Neon

- Serverless-драйвер нативно дружит с Vercel Functions (нет постоянного соединения).
- Интеграция с Vercel в одну кнопку из dashboard.
- Бесплатного tier хватит для одного юзера.

## Auth: Auth.js v5

- Clerk отпал: хотим разобраться в auth самостоятельно.
- Auth.js v5 — опенсорс, Drizzle-адаптер есть, хорошая интеграция с Next.js App Router.
- userId всегда берётся из сессии Auth.js, передаётся явно в lib/.
- Планируется шаринг между юзерами → нужна authorization поверх auth (отдельная задача).

## Валидация: Zod

- Используется в Server Actions для валидации входных данных.
- В lib/ не используется — туда приходят уже валидные данные.

## Форматирование: Prettier

- create-next-app не включает Prettier, добавляем вручную.

## Тесты: Vitest

- Тестируем только `lib/` — чистые функции, легко покрываются unit-тестами.
- Компоненты и Server Actions не тестируем (pet-проект, не продакшн).

## Package manager: pnpm

- Быстрее npm, строгий резолвер зависимостей, рекомендован для Next/Vercel.

## Стили: Tailwind + shadcn/ui

- Mantine отпал: все компоненты клиентские, мешает изучению RSC.
- shadcn/ui — компоненты копируются в проект (твой код, не node_modules), RSC-совместимо.
- Tailwind: ИИ хорошо генерирует, стандарт в Next.js экосистеме.
- Стиль shadcn: `new-york` (переключились с `base-nova`) — лучше подходит по визуальному языку.

## Модель данных: транзакции принадлежат кошельку, не юзеру

- Сущность `wallet` + `wallet_members` (owner/member) вместо прямой привязки транзакций к юзеру.
- Позволяет шарить кошелёк между людьми без переписывания логики.
- Юзер создаётся автоматически при первом входе с ролью owner.
- `createdBy` на транзакции — для аудита, не для доступа.

## Auth: стратегия JWT, не database sessions

- `session: { strategy: 'jwt' }` — сессия в cookie, не в таблице sessions.
- Причина: нет lookup в БД на каждый запрос; Neon serverless не любит частые короткие соединения.
- Минус: нельзя инвалидировать сессию на сервере мгновенно. Для pet-проекта приемлемо.

## Хранение сумм: копейки (integer)

- `amount` в БД хранится в копейках (×100), наружу отдаётся в рублях.
- Custom Drizzle type с `fromDriver(val / 100)` / `toDriver(val * 100)`.
- Причина: float-арифметика ненадёжна для денег.

## Инвайты: UUID-токен без срока действия

- Ссылка-приглашение = случайный UUID, хранится в `wallet_invites`.
- Без expiry — упрощение для pet-проекта. Отзывается вручную.
- Только owner кошелька может создавать и отзывать инвайты.
- После принятия инвайта — редирект обратно (сохраняется через `callbackUrl`).

## PWA: ручной service worker, без next-pwa

- Реализован вручную (`public/sw.js`) вместо `next-pwa`.
- Причина: контроль над стратегией кеширования, нет чёрного ящика.
- RSC-ответы (клиентская навигация Next.js) не кешируются — они динамические и давали стейл-данные.
- Кешируются только: статика (иконки, картинки, манифест) и последний HTML navigate-ответа (офлайн-фолбэк).

## HTTPS в dev

- `next dev --experimental-https` — нужен для регистрации service worker и `secure` cookies.
- Без HTTPS браузер не активирует SW и не принимает httpOnly+secure куки в localhost.

## Гостевой режим: httpOnly cookie вместо localStorage

- `guest-token` хранится в httpOnly cookie (`maxAge: 1 год`), не localStorage.
- Причина: iOS PWA (домашний экран) имеет изолированный localStorage от Safari — токен пропал бы при переходе. Cookie общая для всех контекстов одного origin.
- Авторизация через тот же Credentials-провайдер Auth.js с полем `guestToken`.
- `isGuest` пробрасывается в JWT → session. Шаринг и управление участниками скрыты для гостей.

## Порядок разработки: сначала CRUD, потом auth

- В M1 вместо реального userId использовали `DEV_USER_ID = 'dev-user'` хардкодом.
- Причина: добавлять auth до того, как работает основной поток — лишняя когнитивная нагрузка и риск.
- Переход к реальному auth в M2: заменили константу на `requireUserId()` из сессии Auth.js.
- Принцип: сначала докажи, что логика работает, потом добавляй авторизацию.

## Тесты: реальная БД, не моки

- Integration-тесты в `lib/*.test.ts` ходят в настоящую Neon БД через HTTP-драйвер.
- Причина: мок БД не проверяет реальное поведение (типы, constraints, Drizzle-маппинг копеек).
- Тесты сами создают данные в `beforeEach` и удаляют в `afterEach`.
- Dev-ветка Neon изолирует тестовые данные от продакшн.

## Тёмная тема: class-based + inline script

- Тема управляется классом `dark` на `<html>`, а не только `@media prefers-color-scheme` — так переключение работает независимо от ОС.
- Предпочтение (`light | dark | system`) хранится в `localStorage`; `useTheme` компонуется поверх `useLocalStorage`.
- Inline `<script>` в `layout.tsx` читает `localStorage` синхронно до рендера и вешает класс `dark` — устраняет мерцание белого экрана при загрузке.
- `suppressHydrationWarning` на `<html>`: сервер не знает предпочтений пользователя, расхождение намеренное.
- `color-scheme: light` на `:root` и `color-scheme: dark` на `.dark` — браузер сам переключает нативные контролы (date picker, скроллбар).

## Сканирование чеков: Claude Vision, не OCR

- Использован `claude-sonnet-4-6` через `@anthropic-ai/sdk`.
- Альтернативы отброшены: Tesseract/Google Vision требуют дополнительного парсинга; Veryfi/Mindee — ещё один платный сервис с передачей финансовых данных.
- Промпт явно перечисляет ключевые слова итога (УКУПНО, Готовина, ИТОГО) и запрещает брать строки НДС/ПДВ — без этого модель путала налог с итогом.
- HEIC конвертируется в JPEG на клиенте через `heic2any` (динамический импорт — библиотека обращается к `window` и не работает в SSR).
- Лимит сканирований хранится в БД (`receipt_scan_limit` на юзера, default 5/день) — консольный spending limit Anthropic слишком грубый.
- Клиентское сжатие: Canvas API, max 1200px по длинной стороне, JPEG 85%.

## Persistence UI-предпочтений: cookie, не localStorage

- Валюта и тип транзакции, валюта отображения хранятся в cookie, а не в localStorage.
- Проблема localStorage: сервер его не видит → SSR рендерит дефолт → браузер красит дефолтный HTML до гидрации → видимая вспышка при переключении на сохранённое значение. Клиентские приёмы (lazy-init `useState`, `useLayoutEffect`, `dynamic ssr:false`) не лечат: серверный HTML первичен, а `ssr:false` ещё и ломает скелетоны (pop-in).
- Cookie уходит с HTTP-запросом → сервер читает её в `WalletBody` и засевает `CookiesProvider` → первый же рендер (в т.ч. серверный рендер клиентского компонента) корректный, без вспышки.
- Whitelist через явные `.get()` по трём именам, не `getAll()`: `cookies()` на сервере читает и httpOnly-cookie, поэтому `getAll()` протащил бы сессионный токен Auth.js в клиентский контекст (RSC-пейлоад + `document.cookie`-доступ) — утечка.
- `useCookieState` построен на контексте (React 19: `<Context>` как провайдер без `.Provider`, `use()` вместо `useContext`). Провайдер навешивается серверным `WalletBody` снаружи `WalletView`, поэтому начальное значение не пробрасывается пропами через дерево — листья читают из контекста.
- Тема осталась на localStorage: там вспышки нет — класс `dark` вешает inline-скрипт в `layout.tsx` до отрисовки.

## Миграции в build

- `drizzle-kit migrate && next build` — миграции применяются перед каждым деплоем на Vercel.
- Причина: Vercel не запускает миграции автоматически; без этого новый код падал с 500 на новых колонках.
