# Expense Tracker — Project Context

## Что это

Личный pet-проект для отслеживания трат и доходов.
Двойная цель: (1) изучить React 19 по-максимуму на реальном кейсе,
(2) заложить bot-ready архитектуру, чтобы позже добавить Telegram-бота
без переписывания логики.

Разработка ведётся в режиме вайбкодинга соло-разработчиком (senior frontend).
Не оверинженерь. Это не продакшн на тысячи юзеров.

## Стек

- Next 16 (App Router)
- React 19 (RSC, Server Actions)
- TypeScript, strict mode
- Drizzle ORM + PostgreSQL (Neon)
- Tailwind CSS + shadcn/ui
- Auth.js v5
- Vitest
- Zod
- Prettier
- pnpm
- Деплой: Vercel (цель — PWA, устанавливается на домашний экран)

## Архитектурные правила (НЕ нарушать)

- Вся доменная логика (работа с тратами/доходами, обращения к БД) живёт
  в `lib/` как ЧИСТЫЕ функции: `createTransaction(userId, data)`,
  `getTransactions(userId, filters)`, `getStats(userId, period)` и т.п.
- Server Actions — ТОЛЬКО тонкие обёртки `'use server'` поверх функций из `lib/`.
  В экшене никакой бизнес-логики, только валидация входа + вызов lib + revalidate.
- Компоненты НЕ содержат бизнес-логику и не ходят в БД напрямую.
- Каждая транзакция привязана к `userId`. userId приходит из сессии Auth.js,
  в lib/ всегда передаётся явно. Планируется шаринг между юзерами.
- React 19 фичи использовать осознанно там, где уместно:
  `useActionState` (состояние формы), `useOptimistic` (мгновенное добавление в список),
  `useFormStatus`, `use()`. Не лепить ради галочки.

## Чего НЕ делать

- Не тащить логику работы с данными в компоненты или прямо в Server Actions.
- Не плодить API-роуты (`app/api/...`) без явной необходимости — есть Server Actions.
  Исключение: вебхук для будущего Telegram-бота (когда дойдём).
- Не добавлять npm-зависимости без явного запроса. Сначала предложи, потом ставь.
- Не рефакторить то, о чём не просили.
- Не генерировать большие куски «на будущее» — только то, что нужно сейчас.

## Структура

```
/CLAUDE.md           — этот файл
/docs/decisions.md   — лог архитектурных решений (почему PWA, почему Drizzle и т.д.)
/lib/                — чистая доменная логика (ядро, переиспользуется ботом)

  stats.ts           — агрегации/статистика
  db.ts              — клиент Drizzle (подключение)
  schema.ts          — схема таблиц Drizzle
/app/                — Next App Router: страницы, Server Actions, UI-компоненты
```

## Как мы работаем (режимы)

Один промпт = один режим. Не смешивать.

- **ПЛАН**: обсуждаем подход и затронутые файлы, код не пишем.
- **РЕАЛИЗАЦИЯ**: узкая задача на ОДИН слой (lib → action → UI), отдельный коммит.
- **РЕВЬЮ**: критикуем готовый дифф на соответствие этим правилам.

Фичи режем вертикально по слоям, по одному за раз, коммит после каждого
проверенного куска.

## Definition of done для куска

Перед каждым коммитом обязательно запустить все четыре команды:

```bash
pnpm prettier --write .
pnpm eslint . --fix
pnpm tsc --noEmit
pnpm test
```

- TS strict без ошибок, ESLint чистый.
- Логика — в lib/, не протекла в UI/actions.
- userId передаётся явно.
- Новый файл в `lib/` → рядом `lib/*.test.ts` с тестами на публичные функции.
- Новая страница с серверным `await` (fetch / запрос в БД) → рядом `loading.tsx` со скелетоном.
- Закоммичено с осмысленным сообщением.
